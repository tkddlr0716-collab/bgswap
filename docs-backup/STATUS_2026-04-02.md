# BgSwap 프로젝트 현황 — 2026-04-02

## 1. 서비스 개요

- **URL**: https://bgswap.io
- **요약**: 상품 사진 → AI 배경 제거 → 15가지 배경 자동 합성 SaaS
- **타겟**: Amazon/Etsy/Shopify/eBay 중소 셀러 (50~500장 일괄 처리)
- **가격**: 무료 1장 프리뷰 / Starter $4.99 (10장) / Pro $29 (100장)

- Replicate 하드웨어 비공개이나, T4~L40S 범위에서 **$0.0004~$0.002/장** 추정
- 보수적으로 **$0.002/장** (L40S 기준) 적용

> 마진이 극히 높으므로, 향후 더 비싼 고품질 모델(예: gpt-image-1)로 교체해도 가격 유지 가능.

## 4. Phase D 구현 상세 (2026-04-02)

### 핵심 목표

GPT 무료 1장 품질에 열세 → **대량 자동화(50~500장)**가 유일한 차별점.
기존 1장→결제→처리 플로우를 **결제→100장 업로드→3장 병렬 자동 처리** 구조로 전환.

### 아키텍처: 폴링 + Cron 기반 병렬 처리

```
[클라이언트]                        [서버]

1장씩 업로드 ──POST──→  /api/upload (orderId 포함, 1장씩)
   ×100회                  ↓ images row 생성 (process_status='pending')

결제 완료 ←── webhook ──  Polar
                          ↓ orders.plan = 'starter'|'pro'
                          ↓ pending 이미지 최대 3장 → /api/generate-one 동시 트리거

폴링 ──GET──→  /api/status/{id}           Cron (5분 간격)
   3초 간격              ↓                       ↓
                    slots = 3 - processing    stalled 주문 탐지
                    slots > 0이면:            → generate-one 직접 트리거
                    → pending 최대 3장 트리거   (브라우저 없이도 동작)
                         ↓
                    /api/generate-one (1장, ~29초)
                    → Replicate 배경제거 (~2초)
                    → Sharp 15배경 합성 (Promise.all)
                    → R2 16파일 업로드 (Promise.all)
                    → DB batch INSERT
                    → process_status → 'done'
                    전부 done이면 order.status → 'completed'
```

### 이 아키텍처의 한계와 대응

| 한계 | 심각도 | 현재 대응 | 비고 |
|------|--------|----------|------|
| 브라우저 닫으면 처리 중단 | 높음 → **완화** | cron 하루 1회 캐치업 + 이메일 (Hobby 제약). 폴링이 주 트리거. | Pro 플랜 시 5분 간격 가능 |
| 100장 처리 시간 | 중간 → **개선** | 파이프라인 병렬화(47→29초/장) + 3장 동시 처리 → **~16분** | 실측 기반, 남은 병목은 Sharp 2048px CPU 바운드 |
| 동시 탭 폴링 시 중복 API 호출 | 낮음 | generate-one의 atomic UPDATE로 실제 중복 처리 방지 | 불필요한 호출은 발생 가능 (409 반환) |

> 부수효과: 폴링 없으면 처리 안 함 → 결제만 하고 안 쓰면 비용 0. 이는 장점이 아니라 아키텍처 한계의 부수 효과. 서버사이드 큐 도입 시 사라짐.

### 변경 파일 상세

#### DB 스키마 (`src/lib/db.ts`)
- `orders.plan TEXT` — starter/pro (무료 사용자는 null, upload API에서 `plan === null`이면 1장 제한)
- `orders.resume_email_sent_at TEXT` — 재개 이메일 발송 시각 (전용 컬럼, options JSON과 분리하여 concurrent write 방지)
- `images.process_status TEXT` — pending/processing/done/failed
- `images.retry_count INTEGER DEFAULT 0` — 재시도 횟수 (외부 최대 3회)
- `images.processing_started_at TEXT` — processing 시작 시각 (stuck 감지용, 90초 초과 시 자동 리셋)
- `idx_images_process_status` 인덱스

#### 업로드 API (`src/app/api/upload/route.ts`)
- `orderId` 파라미터 → 기존 결제 완료 주문에 이미지 추가
- plan 기반 동적 제한: pro=100, starter=10, null(무료)=1
- 무료 업로드: `existingOrderId` 없을 때 `files.length > 1` 차단
- INSERT 시 `process_status='pending'`
- **Rate limit 이중화**: 무료=3회/분(abuse 방지), 결제 후 bulk=120회/분(100장 ~50초 가능)
  - orderId 유효성(DB 존재 여부) + paid 상태 검증을 **먼저** 수행한 후에만 120회/분 적용
  - 가짜 orderId로 relaxed limit을 탈 수 없음

#### 1장 처리 API (`src/app/api/generate-one/route.ts`) — 신규
- `pending → processing` atomic UPDATE + `processing_started_at` 기록 (rowsAffected 확인, 0이면 409)
- **파이프라인 병렬화** (47초 → 29초, -39%):
  - Step 1: Replicate 배경 제거 (~2초)
  - Step 2: Sharp 합성 15배경 `Promise.all` 병렬 (기존: 순차 for 루프)
  - Step 3: R2 업로드 16파일 `Promise.all` 병렬 (기존: 1건씩 순차)
  - Step 4: DB INSERT 16건 `db.batch()` 일괄 (기존: 1건씩 순차)
- **Replicate 타임아웃**: 45초 AbortSignal (`replicate.ts`), Vercel 60초 함수 제한 내
- **Replicate 내부 재시도**: `removeBackground()` 최대 3회 시도 + 점진적 백오프 (1초→2초)
- **결과 fetch**: 15초 AbortSignal + HTTP status 검증 + 최대 3회 재시도 + 점진적 백오프
- 성공: `done` / 실패: `failed` + try-catch로 반드시 상태 전환
- 모든 upload 이미지 done → `order.status='completed'`

#### Status API (`src/app/api/status/[id]/route.ts`)
- `processedCount`, `processingCount`, `pendingCount`, `failedCount` 응답
- 자동 재시도: failed만 남고 pending/processing 없으면 → retry_count < 3인 것만 `failed→pending` 리셋 + retry_count 증가
- **retry 소진**: retry_count ≥ 3인 failed만 남으면 → `order.status='completed'` (부분 완료)
  - 부분 완료 구분: order 레벨에서 별도 status 없이 `failedCount > 0`으로 판별
  - result 페이지 UI에서 `completed + failedCount > 0`이면 경고 아이콘 + "N개 실패" 메시지 + 성공 수/실패 수 표시
  - 판단 근거: 별도 `completed_with_errors` status 추가보다 `failedCount` 필드가 단순하고, 폴링/트리거 로직에 영향 없음
- **Stuck processing 복구**: processing이 90초 초과 시 pending으로 자동 리셋 (`processing_started_at` 기준)
- 폴링 트리거: paid + pending > 0 + `slotsAvailable = 3 - processingCount` > 0 → 부족분만큼 generate-one 동시 호출

#### Webhook (`src/app/api/webhook/polar/route.ts`)
- Plan 판별: checkout URL의 `metadata[plan]=starter|pro`를 **우선** 사용, 없으면 상품명/금액 fallback
- 기존 `/api/generate` 전체 루프 → 첫 pending **3장** 동시 `/api/generate-one` 트리거

#### 결제 후 업로드 페이지 (`src/app/order/[id]/upload/page.tsx`) — 신규
- 결제 확인(status=paid) → plan에 따라 10/100장 제한
- 1장씩 순차 업로드 + 개별 진행률 + 자동 재시도(3회)
- 완료 후 `/result/{orderId}`로 이동

#### Result 페이지 (`src/app/result/[id]/page.tsx`)
- Bulk upload CTA: paid + uploadCount ≤ 1이면 `/order/{id}/upload` 링크
- 대량 진행률: 상품 단위 프로그레스 + 예상 잔여 시간
- failed 경고, 확장된 폴링 (pending+processing > 0이면 계속)

#### Cron 통합 (`src/app/api/cron/cleanup/route.ts`)
- Vercel Hobby 플랜 제약: cron 하루 1회만 허용 → cleanup + 자동 처리 + 이메일을 **1개 cron에 통합**
- 매일 03:00 실행, `CRON_SECRET` Bearer 토큰 인증
- **4단계**:
  1. 만료 이미지 R2 삭제 + DB 정리 (기존 cleanup)
  2. Stuck processing 복구 (>90초 → pending 리셋)
  3. Stalled 주문의 pending 이미지 → generate-one 트리거 (PARALLEL_LIMIT=3)
  4. 재개 이메일 발송 (30분 내 미발송 시)
- `/api/cron/resume-notify`는 수동 호출용으로 유지

#### 처리 재개 메커니즘 정리
- **브라우저 열림**: status API 3초 폴링 → 즉시 트리거 (주 경로)
- **브라우저 닫힘**: cron이 하루 1회 캐치업 + 이메일 발송 (Hobby 제약)
- **Pro 플랜 업그레이드 시**: cron을 5분 간격으로 분리하면 브라우저 닫아도 5분 내 자동 재개 가능

#### 이메일 (`src/lib/email.ts`)
- `sendResumeEmail()` 추가

### 동시성 보호 정리

| 위치 | 보호 방법 | 한계 |
|------|----------|------|
| generate-one | `pending→processing` atomic UPDATE + `processing_started_at` 기록, rowsAffected=0이면 409 | 동일 이미지 중복 처리 완전 방지 |
| status API 트리거 | `slotsAvailable = 3 - processingCount` 체크 후 부족분 트리거 | 동시 폴링 시 중복 호출 가능 (generate-one에서 최종 방어) |
| stuck processing 복구 | `processing_started_at < 90초 전`이면 pending으로 리셋 | 정상 처리 중인 것을 리셋할 위험 없음 (Vercel 60초 함수 제한 + Replicate 45초 타임아웃) |
| resume-notify cron | `resume_email_sent_at` 전용 컬럼, SQL 조건으로 필터 | 10분 간격 cron이므로 동시 실행 가능성 낮음 |
